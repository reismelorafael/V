Tática 2: Longo termo (Próximas releases)
Implementar custom backup system:

2.1 Criar EncryptedBackupManager class
```
public class EncryptedBackupManager {
    private static final String BACKUP_KEY_ALIAS = "vectras_backup_key";
    private KeyStore keystore;
    
    public void backupUserData(File outputDir) {
        // 1. Obter symmetric key do KeyStore (hardware-backed se possível)
        SecretKey backupKey = getOrCreateBackupKey();
        
        // 2. Coletar dados críticos
        List<File> filesToBackup = new ArrayList<>();
        filesToBackup.add(getVMImageFile());
        filesToBackup.add(getConfigFile());
        // Excluir: plaintext logs, temporary files, cache
        
        // 3. Criar tar.gz de files
        File tarGz = createTarGz(filesToBackup);
        
        // 4. Encriptar com AES-256-GCM
        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        cipher.init(Cipher.ENCRYPT_MODE, backupKey);
        byte[] iv = cipher.getIV();
        byte[] encrypted = cipher.doFinal(readFile(tarGz));
        
        // 5. Salvar IV + encrypted data
        // IV é public (não é secret), encrypted data requer key
        Files.write(outputDir.resolve("backup.enc"), encrypted);
        Files.write(outputDir.resolve("backup.iv"), iv);
        
        // 6. Não exportar backup para Android's backup system
        // User pode fazer manual export se deseja, mas é encriptado
    }
    
    public void restoreUserData(File backupDir) {
        // Reverse: decrypt, extract tar.gz, restore files
        // Key é stored em KeyStore (não em backup), so sem key = sem restore
    }
    
    private SecretKey getOrCreateBackupKey() {
        // Use KeyStore hardware-backed si disponível (Android 6.0+)
        // Garante chave é stored seguramente, não acessível via adb
        KeyGenerator kgen = KeyGenerator.getInstance("AES");
        kgen.init(256); // AES-256
        return kgen.generateKey();
    }
}
```

2.2 Configurar KeyStore para hardware backing
├─ On Android 6.0+, use AndroidKeyStore provider
├─ Set KeyProperties.PURPOSE_ENCRYPT | PURPOSE_DECRYPT
├─ Set userAuthenticationRequired() = true (requer biometric ou PIN)
└─ Resultado: Key não pode ser exported, even com adb access

2.3 Integrar com app lifecycle
├─ Offer manual export: "Export encrypted backup" button em settings
├─ User pode então salvar arquivo em cloud (Dropbox, Google Drive)
├─ Arquivo é encrypted, então cloud provider não vê dados
└─ Restore: User import arquivo, biometric unlock, data restored

Hiperconsciência: Este approach não cria false sense of security. Mesmo com
encryption, se app é compromised (RCE), attacker pode read data from memory.
Porém, it prevents **passive data extraction** (adb backup). Layered defense.

SOLUÇÃO #3: RACE CONDITION REMEDIAÇÃO - DETERMINISTIC STATE MACHINE
────────────────────────────────────────────────────────────────────────────

Problema BUG #2: ProcessSupervisor field volatile sem proper synchronization.

ESTRATÉGIA: Implement state machine com explicit synchronization e atomic
transitions.

Tática: Rewrite ProcessSupervisor com garantia de thread-safety

3.1 Define state machine explicitamente
```
public class ProcessSupervisor {
    enum State {
        INITIAL,    // Criada mas sem process
        STARTING,   // Tentando start process
        RUNNING,    // Process rodando normalmente
        DEGRADED,   // Output flooding ou performance issue
        STOPPING,   // Tentando stop (graceful)
        STOPPED,    // Process parou com sucesso
        FAILED      // Process morreu unexpectedly
    }
    
    // Use AtomicReference para garantir visibility + atomicity
    private final AtomicReference<State> state = 
        new AtomicReference<>(State.INITIAL);
    
    // Sempre sincronizar STATE TOGETHER com PROCESS
    private final Object stateLock = new Object();
    private Process process;
```

3.2 Implementar thread-safe state transitions
```
public void startProcess() {
    synchronized (stateLock) {
        if (!state.compareAndSet(State.INITIAL, State.STARTING)) {
            throw new IllegalStateException(
                "Cannot start from state: " + state.get());
        }
        try {
            // Start process with security-aware builder
            this.process = createSecureProcessBuilder().start();
            state.set(State.RUNNING);
        } catch (Exception e) {
            state.set(State.FAILED);
            throw e;
        }
    }
}

public void stopProcess() {
    synchronized (stateLock) {
        if (state.get() == State.STOPPED || state.get() == State.FAILED) {
            return; // Already stopped
        }
        
        state.set(State.STOPPING);
        try {
            // Graceful stop via QMP
            sendQMPPowerdown();
            
            // Esperar com timeout
            if (!process.waitFor(5, TimeUnit.SECONDS)) {
                // Timeout: escalate para harder stop
                process.destroy();
                if (!process.waitFor(3, TimeUnit.SECONDS)) {
                    process.destroyForcibly();
                    process.waitFor(2, TimeUnit.SECONDS);
                }
            }
            state.set(State.STOPPED);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            state.set(State.FAILED);
        } finally {
            this.process = null;
        }
    }
}
```

3.3 Implementar state query methods (para outros threads)
```
public synchronized State getState() {
    return state.get();
}

public synchronized boolean isRunning() {
    return state.get() == State.RUNNING;
}

// Nunca permitir external access a process field
// Sempre ir através de state + synchronized methods
```

Hiperlucidez: Problema original era múltiplos threads fazendo operações no
process simultâneamente. Solução não é "adicionar synchronized," é **redesign
para que only um thread controle process**, e outras threads convertem para
isso através de state queries + callbacks.

Tática 4.2: Implementar signature verification
```
public class RomImportReceiver extends BroadcastReceiver {
    private static final String EXPECTED_SIGNER_HASH = 
        "AA:BB:CC:DD:EE:FF:..."; // SHA-256 of app signing cert
    
    @Override
    public void onReceive(Context context, Intent intent) {
        // 1. Verify sender's signature
        String senderPackage = getCallingPackage();
        if (!verifyPackageSignature(context, senderPackage, EXPECTED_SIGNER_HASH)) {
            Log.w(TAG, "Rejecting ROM import from unsigned package: " + senderPackage);
            return;
        }
        
        // 2. Verify file integrity
        Uri romUri = intent.getData();
        if (!verifyROMIntegrity(context, romUri)) {
            Log.w(TAG, "ROM file failed integrity check");
            return;
        }
        
        // 3. Safe to proceed
        importROM(context, romUri);
    }
    
    private boolean verifyPackageSignature(Context context, String packageName,
            String expectedHash) {
        try {
            PackageManager pm = context.getPackageManager();
            PackageInfo info = pm.getPackageInfo(packageName, 
                PackageManager.GET_SIGNATURES);
            
            for (Signature sig : info.signatures) {
                String hash = computeHash(sig.toByteArray());
                if (expectedHash.equals(hash)) {
                    return true;
                }
            }
        } catch (Exception e) {
            Log.e(TAG, "Error verifying signature", e);
        }
        return false;
    }
    
    private boolean verifyROMIntegrity(Context context, Uri romUri) {
        try {
            InputStream is = context.getContentResolver().openInputStream(romUri);
            String romHash = computeHash(is);
            
            // Compare dengan whitelist de known-good ROMs
            Set<String> whitelist = loadROMWhitelist();
            return whitelist.contains(romHash);
        } catch (IOException e) {
            return false;
        }
    }
}
```

SOLUÇÃO #5: COMMAND INJECTION - ARRAY-BASED EXECUTION
────────────────────────────────────────────────────────────────────────────

Problema BUG #4: Runtime.getRuntime().exec(String) permite shell injection.

ESTRATÉGIA: Nunca passar strings simples para exec(). Sempre usar array of
arguments que não são interpreted pelo shell.

Tática 5.1: Criar CommandExecutor abstraction
```
public class SecureCommandExecutor {
    public static class Command {
        private final List<String> arguments;
        private final Map<String, String> environment;
        private final File workingDirectory;
        
        public Command(String executable, String... args) {
            this.arguments = new ArrayList<>();
            this.arguments.add(executable);
            this.arguments.addAll(Arrays.asList(args));
            this.environment = new HashMap<>();
            this.workingDirectory = null;
        }
    }
    
    public static Result execute(Command cmd) throws IOException {
        // Use ProcessBuilder com array of arguments
        ProcessBuilder pb = new ProcessBuilder(cmd.arguments);
        
        // Explicitar environment (não herdar parent)
        pb.environment().clear();
        pb.environment().putAll(createSafeEnvironment());
        
        if (cmd.workingDirectory != null) {
            pb.directory(cmd.workingDirectory);
        }
        
        // Execute
        Process p = pb.start();
        
        // Capture output
        String stdout = readStream(p.getInputStream());
        String stderr = readStream(p.getErrorStream());
        
        int exitCode = p.waitFor();
        
        return new Result(exitCode, stdout, stderr);
    }
    
    private static Map<String, String> createSafeEnvironment() {
        Map<String, String> env = new HashMap<>();
        // Whitelisted safe variables apenas
        env.put("PATH", "/system/bin:/system/xbin");
        env.put("LANG", "en_US.UTF-8");
        env.put("TZ", "UTC");
        // NOT incluindo LD_LIBRARY_PATH, LD_PRELOAD, etc
        return env;
    }
}

// Usage:
SecureCommandExecutor.Command cmd = 
    new SecureCommandExecutor.Command("/system/bin/mount",
        "-t", "tmpfs", "-o", "size=512M",
        "/mnt/tmp");
SecureCommandExecutor.Result result = 
    SecureCommandExecutor.execute(cmd);
```

Hiperlucidez: O ponto crítico é que array-based invocation TIDAK goes através
shell. Shell metacharacters são treated como literal strings. "/system/bin/sh"
não é invoked, então ";" não é command separator - é literal string argument.

SOLUÇÃO #6: JNI BOUNDS VALIDATION - DEFENSIVE PROGRAMMING IN NATIVE CODE
────────────────────────────────────────────────────────────────────────────

Problema BUG #5: nativeCopyBytes() tidak melakukan bounds check pada JNI array.

ESTRATÉGIA: Validasi array bounds SEBELUM any pointer arithmetic.

Tactic 6.1: Rewrite nativeCopyBytes dengan proper bounds checking
```c
JNIEXPORT jint JNICALL Java_com_vectras_vm_core_NativeFastPath_nativeCopyBytes(
    JNIEnv* env, jclass clazz,
    jbyteArray src, jint srcOffset, 
    jbyteArray dst, jint dstOffset, 
    jint length) {
    
    // 1. Null check
    if (!src || !dst) return -1;
    
    // 2. Offset dan length sanity check
    if (srcOffset < 0 || dstOffset < 0 || length < 0) return -2;
    
    // 3. GET ARRAY LENGTHS
    jsize srcLen = (*env)->GetArrayLength(env, src);
    jsize dstLen = (*env)->GetArrayLength(env, dst);
    
    // 4. BOUNDS CHECKING SEBELUM ARITHMETIC
    // Check: srcOffset + length <= srcLen
    if (srcOffset > srcLen) return -3;  // srcOffset itself out of bounds
    if (length > srcLen - srcOffset) return -4;  // length would overflow
    
    // Check: dstOffset + length <= dstLen
    if (dstOffset > dstLen) return -5;  // dstOffset itself out of bounds
    if (length > dstLen - dstOffset) return -6;  // length would overflow
    
    // 5. NOW IT'S SAFE - get pointers
    jbyte* s = (*env)->GetPrimitiveArrayCritical(env, src, NULL);
    if (!s) return -7;
    
    jbyte* d = (*env)->GetPrimitiveArrayCritical(env, dst, NULL);
    if (!d) {
        (*env)->ReleasePrimitiveArrayCritical(env, src, s, JNI_ABORT);
        return -8;
    }
    
    // 6. SAFE COPY (pointers are guaranteed in-bounds)
    memcpy((uint8_t*)d + dstOffset, (uint8_t*)s + srcOffset, length);
    
    (*env)->ReleasePrimitiveArrayCritical(env, src, s, JNI_ABORT);
    (*env)->ReleasePrimitiveArrayCritical(env, dst, d, 0);
    
    return 0;  // Success
}
```

SOLUÇÃO #7: MEMORY LEAK REMEDIAÇÃO - LIFECYCLE CLEANUP FRAMEWORK
────────────────────────────────────────────────────────────────────────────

Problemas BUG #6,#16,#17,#21-24: Static references, handlers, thread leaks.

ESTRATÉGIA: Implement central Lifecycle Manager que tracks todos resources
dan ensures cleanup di appropriate lifecycle point.

Tactic 7.1: Criar ResourceLifecycleManager
```
public class ResourceLifecycleManager {
    private static final List<Cleaner> cleaners = 
        new CopyOnWriteArrayList<>();
    
    @FunctionalInterface
    public interface Cleaner {
        void clean() throws Exception;
    }
    
    public static void register(Cleaner cleaner) {
        cleaners.add(cleaner);
    }
    
    public static void cleanupAllResources() {
        // Called dari Activity.onDestroy(), Service.onDestroy()
        for (Cleaner cleaner : cleaners) {
            try {
                cleaner.clean();
            } catch (Exception e) {
                Log.e("ResourceManager", "Error in cleanup", e);
            }
        }
        cleaners.clear();
    }
}
```

Tactic 7.2: Registrar resources dalam Activity
```
public class MainActivity extends AppCompatActivity {
    private Handler handler;
    private Thread worker;
    
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);
        
        // Create handler
        handler = new Handler(Looper.getMainLooper());
        
        // Create thread
        worker = new Thread(() -> {
            // work
        });
        worker.start();
        
        // REGISTER CLEANUP
        ResourceLifecycleManager.register(() -> {
            handler.removeCallbacksAndMessages(null);
        });
        
        ResourceLifecycleManager.register(() -> {
            if (worker.isAlive()) {
                worker.interrupt();
                try {
                    worker.join(5000);
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                }
            }
        });
    }
    
    @Override
    protected void onDestroy() {
        super.onDestroy();
        ResourceLifecycleManager.cleanupAllResources();
    }
}
```
