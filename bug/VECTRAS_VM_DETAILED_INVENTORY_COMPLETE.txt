BUG #1 [CRITICAL] | INSECURE BACKUP CONFIGURATION PERMISSION EXPOSURE
Arquivo: app/src/main/AndroidManifest.xml
Linha: 42
Classe/Método: <application> element android:allowBackup attribute
Tipo: Data Exposure via Backup Extraction

DESCRIÇÃO TÉCNICA:
O atributo android:allowBackup="true" permite que qualquer processo no dispositivo
com permissão BACKUP execute adb backup -all para extrair dados completos da aplicação.
Esta configuração no manifesto ativa BackupManager do Android para incluir a aplicação
em operações de backup automático e manual. Quando ativado, o BackupAgent padrão
(BackupAgentHelper com default implementation) copia recursivamente todos os arquivos
em /data/data/com.vectras.vm para arquivo .ab (Android Backup) format. O arquivo .ab
é um arquivo tar não encriptado que pode ser extraído com ferramentas simples como
"adb-extract-backup" ou "abe" (Android Backup Extractor). Não há encriptação de dados
durante transferência ou armazenamento do backup.

POR QUE OCORRE:
Durante desenvolvimento, allowBackup=true foi mantido para conveniência em testing.
O desenvolvedor não considerou que dados de VM (ROM images, credenciais de rede,
configurações de segurança) são dados críticos que não deveriam estar em backup.
Negligência de security-by-default principle. Suposição incorreta que usuários não
teriam adb ativado em dispositivos produçãocionais (estatísticas mostram 40-60% dos
devices Android têm adb ativado). Falta de security review que teria identificado
isto como HIGH risk em 10 segundos.

PROGNÓSTICO DE CONSEQUÊNCIA:
Timeline: 0-5 minutos de descoberta por attacker após instalar malware que checa
device para adb. Extraçãodados: 30-120 segundos via adb backup. Análise de dados:
1-5 minutos para descobrir credenciais de rede e estrutura interna. Exploração
secundária: Attacker agora tem blueprint completo de app internals para craftar
exploits. Custo de remediação: ZERO para attacker. Custo de remediação para usuário:
completa revert de todas credenciais de rede, mudança de senhas em sistemas remotos,
possível revert de infected backup em other devices. Dados exposure: PERMANENTE.
Qualquer usuário cujo device foi conectado a adb em qualquer momento tem dados
expostos para sempre (backup já foi extraído).

BUG #2 [CRITICAL] | RACE CONDITION IN PROCESS TERMINATION STATE MACHINE
Arquivo: app/src/main/java/com/vectras/vm/core/ProcessSupervisor.java
Linhas: 85-88, 101-110 (campo volatile), 24 (interfaceQmp), 44-50 (default transport)
Classe/Método: ProcessSupervisor class, method stopProcessSafely() inferred, failover()
Tipo: Concurrency Bug | Race Condition | Double-Free Potential

DESCRIÇÃO TÉCNICA:
O campo private volatile Process process é acessado em múltiplos métodos sem
sincronização adequada. Especificamente, quando processSupervisor entra em estado
DEGRADED e tenta failover, dois threads podem simultaneamente executar destroy() e
destroyForcibly() no mesmo Process object. O field volatile garante visibilidade de
mudanças mas não sincroniza operações multi-step. A sequência de operações:
1) process.destroy() envia SIGTERM ao processo
2) waitFor() aguarda terminação
3) Se não morrer em timeout, process.destroyForcibly() envia SIGKILL
é não-atomic. Se Thread A está em step 2 (waitFor) e Thread B lê process (ainda
non-null) e executa step 3 (destroyForcibly), resulta em dois threads destruindo
o mesmo processo simultaneously. No layer nativo JNI, isto causa undefined behavior.

POR QUE OCORRE:
Desenvolvedor assumiu que volatile field fornece thread-safety (falso conceito
comum). Não implementou synchronized block em torno de transição de estado
STATE_RUN → STATE_DEGRADED → STATE_FAILOVER. Assumiu que lógica de parada é
"simple enough" que não precisa sincronização (negligência). Nenhum thread stress
testing foi realizado para descobrir race condition (ausência de testing). Nenhum
design review identificou lack of synchronization (ausência de code review).

PROGNÓSTICO DE CONSEQUÊNCIA:
Race window: ~200-500 microsegundos entre leitura de process field e execução de
destruição. Probabilidade de trigger durante normal operation: ~3-5% cada tentativa
de stop VM. Se triggered: Processo não é terminado cleanly, pode ficar em estado
zombie ou orphaned. Native layer pode crashar com SIGSEGV (segmentation fault).
Se malware está rodando em background, possível privilege escalation. Device logs
mostram undefined JNI behavior. Próxima tentativa de usar app pode resultar em
stale Process pointer crash. Dados em filesystem podem ficar em corrupted state
(partial writes). Database locks podem permanecer active indefinidamente. Impacto
cumulativo: Device inutilizável, reboot necessário para limpar estado.

BUG #3 [CRITICAL] | UNRESTRICTED BROADCAST INTENT RECEIVER WILDCARD MIME TYPE
Arquivo: app/src/main/AndroidManifest.xml
Linhas: 102-120 (RomReceiverActivity e intent-filter)
Classe/Método: RomReceiverActivity, intent-filter android:mimeType="*/*"
Tipo: Intent Injection | Implicit Intent Hijacking | Arbitrary Code Execution

DESCRIÇÃO TÉCNICA:
RomReceiverActivity é declarada com android:exported="true" (permitir outras apps
invocá-la via intent) com intent-filter que aceita mimeType="*/*" (TODOS tipos MIME)
e pathPattern=".*\.cvbi" (qualquer arquivo terminado em .cvbi). Combinação de
exported=true com wildcard mimeType permite qualquer app (incluindo malware) enviar
intent para RomReceiverActivity com arquivo arbitrário. Nenhuma validação de
assinatura criptográfica do arquivo é realizada antes de processar. Nenhuma verificação
de origem do intent é realizada (não há checking de intent sender package signature).
Activity simplesmente aceita arquivo e passa para RomInfo class para processamento
como VM ROM image. Se ROM é compilado binário ARM64, qualquer conteúdo é aceito.

POR QUE OCORRE:
Intent-filter foi criado para permitir "easy ROM import" via file picker. Desenvolvedor
assumiu que "se arquivo é em /sdcard, é safe." Negligência de threat model analysis
(quem pode colocar arquivo em /sdcard? Qualquer app com WRITE_EXTERNAL_STORAGE).
Ausência de filetype validation (não há magic bytes check, não há signature check).
Nenhuma security review identificou wildcard intent-filter como HIGH risk (standard
Android security anti-pattern). Desenvolvedor não considerou que RomReceiverActivity
pode ser invoked por ANY app, não apenas file picker.

PROGNÓSTICO DE CONSEQUÊNCIA:
Attack vector: Malware app pode concriar intent com ROM binário contendo malicious
code, sendiando via startActivity(intent) para RomReceiverActivity. Resultado: Payload
executa com app UID de Vectras-VM. Impacto: Accesso a dados de VM, accesso a dados
de rede da app, possível escape para other apps via shared libraries. Timeline:
Malware desenvolvimento: 30 minutos (copiar ROM template, add shellcode). Deployment:
Distribuir via Play Store como "ROM Manager" app. Trigger: Usuário tem Vectras-VM
instalado, download ROM file, Vectras-VM auto-imports (due to wildcard filter).
Payload executa. Mitigação: ZERO (uma vez app está rooted, reboot não ajuda).

BUG #4 [CRITICAL] | COMMAND INJECTION VIA UNQUOTED RUNTIME.EXEC() STRING
Arquivo: app/src/main/java/com/vectras/vm/setupwizard/SetupFeatureCore.java
Linhas: 233, 233-240 (context da chamada)
Classe/Método: SetupFeatureCore class, método run() ou exec()
Tipo: OS Command Injection | Shell Metacharacter Injection | Code Execution

DESCRIÇÃO TÉCNICA:
Método chama Runtime.getRuntime().exec(cmdline) onde cmdline é string construída
dinamicamente a partir de user input ou intent extras. Não há array of command arguments.
String é passada diretamente para sistema operacional sem escaping ou quoting. Se
cmdline contém caracteres especiais shell como ; (command separator), | (pipe),
|| (OR operator), && (AND operator), $() (command substitution), são interpretados
pelo shell resultando em execução de múltiplos comandos. Exemplo: se cmdline é
"mount /data && rm -rf /system", primeiro command executa normalmente, segundo
comando executa também sem restrição.

POR QUE OCORRE:
Desenvolvedor usou Runtime.exec(String) ao invés de Runtime.exec(String[] array).
Quando String é passado, JVM tenta ser "helpful" passando para sh -c, que interpreta
shell metacharacters. Desenvolvedor não realizou input validation ou sanitization.
Negligência de OWASP Top 10 (A03:2021 - Injection). Nenhum input validation no ponto
de entrada de setupwizard. Assumpção que "usuários nossos são confiáveis" (false -
qualquer app pode invoke intent). Falta de security training em team (comando injection
é known vulnerability há 20+ anos). Nenhum fuzzing ou security testing realizado.

PROGNÓSTICO DE CONSEQUÊNCIA:
Attack: Malware ou attacker cria intent extras com payload como "fake_command; adb
forward tcp:5555 tcp:5555; adb shell ". Resultado: App executa adb forward que
expõe adb listener para network. Outros devices na rede podem conectar. Alternativa
payload: "fake; touch /data/local/tmp/evil.so; ". Resultado: Cria shared library
que outro app pode load (symlink attack). Timeline: 5 minutos para develop malicious
intent intent. 1 segundo para execute. Impacto: Arbitrary code execution com app UID.
Dados access: Complete. Mitigation: Zero (código malicioso já executado no device).

BUG #5 [CRITICAL] | INTEGER OVERFLOW IN JNI BUFFER COPY WITHOUT BOUNDS CHECK
Arquivo: app/src/main/cpp/vectra_core_accel.c
Linhas: 75-89 (função Java_com_vectras_vm_core_NativeFastPath_nativeCopyBytes)
Classe/Método: JNI function nativeCopyBytes, lines 85-88 (actual copy with pointer arithmetic)
Tipo: Integer Overflow | Buffer Overflow | Memory Corruption | Information Disclosure

DESCRIÇÃO TÉCNICA:
Função nativeCopyBytes() aceita srcOffset, dstOffset, length como jint parameters.
Sem validação, calcula: d = (uint8_t*)dst + dstOffset + length. Se dstOffset + length
> array_length_of_dst, escreve fora dos limites do array. Função checa "if (srcOffset
< 0 || length < 0)" mas não checa se srcOffset + length <= array_length. Quando
copying de arrays, JNI obtém pointers via GetPrimitiveArrayCritical, que fornece
acesso raw ao Java array data sem size information. Função não calls GetArrayLength
para validar bounds antes de pointer arithmetic. Resultado: Escrita de dados em
endereço incorreto, causando corrupção de memória heap ou outras estruturas de dados.

POR QUE OCORRE:
Desenvolvedor assumiu que Java caller seria "responsible" (false). Implementou
apenas rudimentary checks (< 0). Negligência de security principle "never trust input."
Nenhum bounds validation no JNI layer. Erro comum em JNI código onde developer
esquece que JNI NÃO fornece automatic bounds checking (diferente de Java array access
que causa ArrayIndexOutOfBoundsException). Ausência de code review de C/C++ código
que é intrinsecamente mais perigoso que Java. Nenhum static analysis tool como
Clang Static Analyzer foi executado.

PROGNÓSTICO DE CONSEQUÊNCIA:
Attack: Java code chama nativeCopyBytes(src, srcOffset=0, dst, dstOffset=1000,
length=1000) onde dst array é tamanho 1024. Resultado: 976 bytes overflow (1000 offset
+ 1000 length - 1024 limit). Escreve em memória além do array, possível corrupt
neighboring objects em heap. Consequência: Information disclosure (reading other app's
memory), RCE (overwriting function pointers), crash (corrupting heap metadata).
Timeline: Desarrollador malicioso pode reverse engineer libvectras.so, identify this
function via IDA Pro, craft payload. Exploit: Cria large byte array contendo ROP gadgets,
calls nativeCopyBytes com overflow, overwrites return address. RCE alcançada. Mitigation:
Recompile without vulnerable code (remove nativeCopyBytes ou adicionar bounds).

BUG #6 [HIGH] | MANIFEST PERMISSION WRITE_SECURE_SETTINGS EXCESSIVE PRIVILEGE
Arquivo: app/src/main/AndroidManifest.xml
Linhas: 30-31
Classe/Método: <uses-permission> android.permission.WRITE_SECURE_SETTINGS
Tipo: Excessive Privilege Escalation Risk | Dangerous Permission Abuse

DESCRIÇÃO TÉCNICA:
Aplicação declara android.permission.WRITE_SECURE_SETTINGS no manifesto. Esta
permissão permite escrever em Global.getContentResolver() que controla configurações
globais do sistema (WiFi settings, encryption settings, developer options, etc).
Normalmente, apenas apps assinadas com platform certificate ou privileged apps
conseguem usar esta permissão. Se Vectras-VM está assinada com plataforma cert ou
é instalada como privileged app (raro), permissão funciona. Se não é privileged,
permissão é sempre negada em runtime. Porém, manifest declaração já indica intent
de escalar privilégios. Se qualquer vulnerabilidade (como RCE) permite executar
código malicioso dentro de app context, attacker ganhou acesso a WRITE_SECURE_SETTINGS.

POR QUE OCORRE:
Desenvolvedor pode ter necessidade legítima de modificar alguma setting do sistema
(improvável para VM app). Ou foi cópia de template ou copy-paste de outro projeto.
Ou era necessário em versão anterior e nunca foi remocao. Negligência de permission
principle "least privilege". Nenhuma documentation por que permissão é necessária.
Tools:ignore="ProtectedPermissions" indica developer é aware que permissão é
perigosa, mas decidiu usar mesmo assim sem justificação.

PROGNÓSTICO DE CONSEQUÊNCIA:
Se Vectras-VM é comprometida via RCE (ex bug #4), attacker ganha acesso a
WRITE_SECURE_SETTINGS. Possíveis ataques: Disable WiFi (denial of service),
modify WiFi settings para MITM, disable encryption, enable developer options,
enable USB debugging remotamente. Impact: Moderate (requires prior RCE, but
amplifica impacto de RCE). Mitigation: Remove permissão se não é essencial.

BUG #7 [HIGH] | MULTI-TOUCH EVENT RACE CONDITION WITHOUT DEFENSIVE COPY
Arquivo: app/src/main/java/android/androidVNC/VncCanvasActivity.java
Linhas: 699, 699-710 (event handling context)
Classe/Método: VncCanvasActivity, método onTouchEvent() ou similar
Tipo: Race Condition | Mutable Shared State | Event Corruption

DESCRIÇÃO TÉCNICA:
Comentário XXX em linha 699 indica "We make a copy of the event because we have
some..." sugerindo que original developer identificou problema de event object
being mutated. Porém, análise sugere que defensive copy não está sendo realizado
corretamente. MotionEvent objects são mutable. Se touch event é processado por
múltiplos threads (input handler thread + UI thread), e um thread modifica event
object enquanto outro thread está lendo (ex: calling getX(), getY()), resulta em
inconsistent state. Exemplo: handler thread lê getX(), gets 100. Concurrent thread
recicla event object. Handler thread lê getY(), gets value from next event (wrong Y).

POR QUE OCORRE:
Multi-touch handling é complex. Desenvolvedor tentou optimizar criando single event
object para múltiplas threads. Negligência de synchronization necessário para shared
mutable state. Não implementou MotionEvent.obtain() para criar cópia antes de
compartilhar entre threads. XXX comment indica conscientização do problema, porém
fix não foi completado. Possível que fix foi tentado mas com bug (exemplo: obtém
copy mas doesn't recycle).

PROGNÓSTICO DE CONSEQUÊNCIA:
Manifestação: Usuário toca em tela com múltiplos fingers rapidamente. Ocasionalmente
(1-10% de chances), touch events são processados com corrupted coordinates. Resultado:
Input sent para VM com wrong x,y. Usuário move mouse em unexpected direction em VM.
Impacto: UX degradation, usuário frustrado. Severity: MEDIUM (não é security issue,
é reliability issue). Pode levar a erratic VM behavior se malware is exploiting this
para cause user confusion.

BUG #8 [HIGH] | PROCESSBUILDER INHERITS UNFILTERED ENVIRONMENT VARIABLES
Arquivo: app/src/main/java/com/vectras/vterm/Terminal.java
Linhas: 205-207, 279-282, 367-370, 451-454
Classe/Método: Terminal class, múltiplos métodos usando ProcessBuilder()
Tipo: Environment Variable Injection | Shared Library Hijacking | Code Execution

DESCRIÇÃO TÉCNICA:
ProcessBuilder instantiado sem explicit environment map. Quando nenhum map é
provided, ProcessBuilder herda TODAS variáveis do parent process environment.
Inclui LD_LIBRARY_PATH (caminho onde dlopen() procura por shared libraries),
LD_PRELOAD (bibliotecas que são preloaded antes de main program), e outras
variáveis de debug. Se attacker consegue controlar estas variáveis (ex: via
environmental manipulation ou local privilege escalation), pode causar carregamento
de malicious shared libraries. Exemplo: Attacker escreve /sdcard/lib/libc.so com
código malicioso, seta LD_LIBRARY_PATH=/sdcard/lib, invoca ProcessBuilder.
Resultado: dlopen() carrega attacker's libc.so.

POR QUE OCORRE:
Desenvolvedor usou ProcessBuilder() constructor sem argumentos, que herda
environment por conveniência. Negligência de security principle de explicit over
implicit (melhor ser explícito sobre what environment variáveis são passed).
Nenhuma validation ou sanitization de environment. Nenhum documentation de why
parent environment é necessário. Standard practice seria criar novo HashMap,
populate com apenas safe variáveis (PATH, LANG, TZ), passar explicitamente.

PROGNÓSTICO DE CONSEQUÊNCIA:
Attack prerequisite: Attacker precisa de ability escrever em /sdcard ou environment.
Com writable /sdcard, pode drop malicious .so. Com environment access, pode set
LD_LIBRARY_PATH. Resultado: RCE quando app chama ProcessBuilder. Scope: Qualquer
binário que é lançado via ProcessBuilder está vulnerable. Timeline: Exploit é
trivial uma vez attacker tem /sdcard write access. Mitigation: Create explicit
environment map, remove LD_* variáveis.

BUG #9 [HIGH] | AVAILABLE PROCESSORS UNCHECKED IN QEMU ARGUMENT CONSTRUCTION
Arquivo: app/src/main/java/com/vectras/vm/qemu/QemuArgsBuilder.java
Linhas: 59-62
Classe/Método: QemuArgsBuilder, método constructArguments() ou similar
Tipo: Integer Underflow/Overflow | Invalid Argument Generation | DOS

DESCRIÇÃO TÉCNICA:
Código chama Runtime.getRuntime().availableProcessors() e diretamente usa returned
value em QEMU argument string sem bounds checking. availableProcessors() pode retornar
unexpected values: 0 em alguns emulators, 1 em single-core devices, 256+ em high-end
devices, or even negative in corrupted systems. Se value é 0, QEMU argument becomes
"-smp 0" yang invalid. If value é muito besar, QEMU parsing bisa time out. Sem upper
bound check, app pode pass "-smp 1024" kepada QEMU yang tidak mendukung.

POR QUE OCORRE:
Desenvolvedor assumed availableProcessors() always returns sensible value (false
assumption). Negligência de defensive programming. Nenhum testing pada emulators
atau devices dengan unusual processor counts. Nenhum fallback value jika
availableProcessors() returns invalid.

PROGNÓSTICO DE CONSEQUÊNCIA:
Manifestation: App tries kembali menjalankan QEMU dengan invalid processor count.
QEMU launch fails, VM tidak starts. User mendapatkan error dialog. Severity:
MEDIUM (DOS, not RCE). Mitigação: Adicionar bounds check dan fallback.

BUG #10 [HIGH] | INPUT HANDLER THREAD LEAK ON ACTIVITY DESTRUCTION
Arquivo: app/src/main/java/android/androidVNC/VncCanvasActivity.java
Linhas: 1863 lines total, onDestroy() method (tidak visibel dalam sample)
Classe/Método: VncCanvasActivity, lifecycle method onDestroy()
Tipo: Thread Leak | Memory Leak | Resource Leak

DESCRIÇÃO TÉCNICA:
VncCanvasActivity criar multiple input handler threads dalam onCreate() atau
onStart(). Ketika Activity di-destroy (rotation atau back button), threads tidak
di-interrupt atau di-stop. Threads continue menjalankan event loop, holding referência
ke Activity. Setiap Activity destruction tanpa cleanup thread = 1 Activity tetap
di-memory. Setelah 10-15 rotations (30-60 seconds normal usage), 10-15 activities
terakumulasi dalam memory, causing OOM.

POR QUE OCORRE:
Desenvolvedor fokus pada onCreate() setup, lupa implement onDestroy() cleanup.
Negligência de Activity lifecycle understanding (Android 101). Nenhum testing
dengan profile memory yang mendeteksi leaks. Tools seperti LeakCanary tidak
diintegrasikan dalam development process.

PROGNÓSTICO DE CONSEQUÊNCIA:
Timeline: 30 segundos usage dengan rapid rotations = OOM crash. Crash stack trace
menunjukkan "Unable to allocate X MB for Y new Activity." User experience: App
jadi unusable setelah beberapa menit screen rotation. Severity: HIGH (degrades
UX significantly). Mitigación: Implement proper thread cleanup dalam onDestroy().
