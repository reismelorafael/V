Ciclo 1: Resource Leaks Não-Óbvios
────────────────────────────────────────────────────────────────────────────

BUG #21 [HIGH] | BufferedReader em BackgroundJob Sem Try-With-Resources
Arquivo: app/src/main/java/com/termux/app/BackgroundJob.java
Linhas: 68, 89
Descrição: BufferedReader criada em linhas 68 e 89 sem try-with-resources ou
finally block. Se exceção ocorre após criação mas antes de close(), reader fica
aberta causando file descriptor leak. Thread lê do stream indefinidamente.

Solução: Envolver em try-with-resources ou implementar try-finally explícito com
close() garantido. Usar try (BufferedReader reader = new BufferedReader(...)) {...}

BUG #22 [HIGH] | Process InputStream Leak em TermuxInstaller
Arquivo: app/src/main/java/com/termux/app/TermuxInstaller.java
Linha: 88
Descrição: BufferedReader lê zipInput (InputStream de processo) mas stream não é
fecha explicitamente se exceção ocorre durante parsing de symlinks. ZipInputStream
permanece aberto consumindo recursos.

Solução: Envolver leitura de zipInput em try-with-resources, garantir close() de
ZipInputStream mesmo em caso de exceção.

BUG #23 [HIGH] | FileInputStream Leak em TermuxFileReceiverActivity
Arquivo: app/src/main/java/com/termux/filepicker/TermuxFileReceiverActivity.java
Linha: 79
Descrição: FileInputStream criada em linha 79 sem try-with-resources. Novo stream
criado a cada iteração sem garantia de fechamento. Em arquivo grande com muitas
iterações, múltiplas FILE descriptors vazam.

Solução: Usar try-with-resources para garantir fechamento automático.

BUG #24 [MEDIUM] | Database Cursor Leak em Vectras (Inferido)
Arquivo: app/src/main/java/com/vectras/vm/rafaelia/HdCacheMvp.java
Tipo: Potencial cursor leak não detectado em análise estática
Descrição: Se HdCacheMvp acessa database (SQLite), qualquer Cursor deve ser
fechado em finally block ou try-with-resources. Cursor não fechado causa memory
leak e database lock.

Solução: Toda query deve usar try-with-resources ou try-finally com cursor.close().

BUG #25 [MEDIUM] | Bitmap Memory Leak em VncCanvas
Arquivo: app/src/main/java/android/androidVNC/VncCanvas.java
Linhas: 2146 (LargeBitmapData ou FullBufferBitmapData)
Descrição: Bitmaps alocados para framebuffer VNC podem não ser reciclados se
Activity é destruída abruptamente. Bitmaps permanecem em native heap causando
OOM gradual.

Solução: Implementar onDestroy() em VncCanvasActivity que chama recycle() em todos
bitmaps. Manter referências de bitmap em list para cleanup bulk.

Ciclo 2: Memory Leaks via Static References e Listeners
────────────────────────────────────────────────────────────────────────────

BUG #26 [HIGH] | VncCanvasActivity Static InputHandler Reference
Arquivo: app/src/main/java/android/androidVNC/VncCanvasActivity.java
Linha: 743
Descrição: public static AbstractInputHandler inputHandler; mantém referência
static à instance de Activity-specific class. Quando Activity é destruída, static
reference previne garbage collection causando memory leak.

Solução: Remover static qualifier, usar WeakReference se compartilhamento é
necessário, ou passar listener via callback interface em lugar de static.

BUG #27 [HIGH] | X11Activity Static Handler Reference
Arquivo: app/src/main/java/com/vectras/vm/x11/X11Activity.java
Linha: 99
Descrição: public static Handler handler = new Handler(); cria referência static
a instance-specific Handler. Memory leak grave se Activity é recreada frequentemente
(rotation, configuration change).

Solução: Remover static, usar local handler ou implementar cleanup em onDestroy().

BUG #28 [HIGH] | MainStartVM Static Handler for Launch
Arquivo: app/src/main/java/com/vectras/vm/main/core/MainStartVM.java
Linha: 55
Descrição: public static final Handler handlerForLaunch mantém handler static.
Callbacks postados para este handler durante Activity lifecycle causam leak.

Solução: Usar HandlerThread com lifecycle management ou remover static.

BUG #29 [HIGH] | VectrasStatus LogListener Vector Memory Leak
Arquivo: app/src/main/java/com/vectras/vm/logger/VectrasStatus.java
Linhas: 25-26, 164-185
Descrição: static Vector<LogListener> e static Vector<StateListener> mantêm
referências a listeners. Se listener registrado em Activity é adicionado à Vector
e não é removido corretamente, Activity fica retida em memory.

Solução: Implementar cleanup explícito em Activity.onDestroy() que remove listeners.
Considerar WeakReference para listeners.

BUG #30 [HIGH] | MainActivity Static Listener Reference
Arquivo: app/src/main/java/com/vectras/vm/main/MainActivity.java
Linha: 107
Descrição: public static CallbackInterface.HomeCallToVmsListener
homeCallToVmsListener; é static e pode reter Activity inteira em memory.

Solução: Implementar WeakReference wrapper ou remover static e passar via
callback interface apropriadamente.

Ciclo 3: Handler e Thread Message Leaks
────────────────────────────────────────────────────────────────────────────

BUG #31 [HIGH] | Panner Handler Postdelayed Leak
Arquivo: app/src/main/java/android/androidVNC/Panner.java
Linha: 73, 89
Descrição: handler.postDelayed(this, 50) onde 'this' é Runnable. Se Activity
é destruída enquanto mensagem aguarda na queue, Activity é retida via Message.target
(Handler) e callback (Panner instance).

Solução: Implementar removeCallbacks em Activity.onDestroy().

BUG #32 [HIGH] | VncCanvas Handler Post Leak
Arquivo: app/src/main/java/android/androidVNC/VncCanvas.java
Linha: 89, 164, 176, 189
Descrição: Múltiplos handler.post() e handler.postDelayed() callsites sem
correspondentes removeCallbacks em cleanup. Messages pendentes retêm Activity.

Solução: Manter referência a Runnables postados, chamar
handler.removeCallbacks(runnable) em onDestroy().

BUG #33 [MEDIUM] | Terminal Handler Post em Dialog
Arquivo: app/src/main/java/com/vectras/vterm/Terminal.java
Linha: 95, 163, 171, 253, 337, 447, 501
Descrição: Múltiplas instâncias de new Handler(Looper.getMainLooper()).post()
dentro de Anonymous Runnables. Se dialog é fechado antes de post executar,
referência circular causa leak.

Solução: Usar Handler local (não novo) ou armazenar e remover callbacks.

BUG #34 [MEDIUM] | BellUtil Handler Postdelayed Leak
Arquivo: app/src/main/java/com/termux/app/BellUtil.java
Linha: 51
Descrição: handler.postDelayed(bellRunnable, MIN_PAUSE...) sem removeCallbacks
correspondente quando objeto é destroyed. Bell thread mantém atividade.

Solução: Implementar cleanup de postdelayed callbacks.

Ciclo 4: Integer Overflow e Boundary Vulnerabilities
────────────────────────────────────────────────────────────────────────────

BUG #35 [HIGH] | Integer Overflow em Bitmap Allocation
Arquivo: app/src/main/java/android/androidVNC/AbstractBitmapData.java
Tipo: Potencial integer overflow em width*height*bytesPerPixel
Descrição: Quando alocando bitmap para display VNC, se width e height são muito
grandes, multiplicação width*height pode fazer integer overflow resultando em
bitmap pequeno alocado para grande região causando buffer overflow.

Solução: Implementar verificação: if (width > MAX_DIMENSION || height >
MAX_DIMENSION || width * height > MAX_PIXELS) throw error.

BUG #36 [MEDIUM] | QemuArgsBuilder CPU Count Saturation Unchecked
Arquivo: app/src/main/java/com/vectras/vm/qemu/QemuArgsBuilder.java
Linhas: 59-62
Descrição: availableProcessors() pode retornar 0 em alguns dispositivos causando
divisão por zero ou argument string vazia. Sem upper bound check, pode resultar
em QEMU command com smp=0 que causa parse error ou crash.

Solução: Implementar bounds check: cpus = Math.max(1, Math.min(cpus, 8))

BUG #37 [MEDIUM] | Array Index Bounds em VNC Color Model
Arquivo: app/src/main/java/android/androidVNC/ColorModel256.java
Tipo: Potencial array out of bounds
Descrição: Se palette array é menor do esperado ou input color value é maior que
array size, acesso array[color] causa ArrayIndexOutOfBoundsException.

Solução: Adicionar bounds check: if (color < 0 || color >= palette.length)
return defaultColor;

Ciclo 5: Path Traversal e File Security
────────────────────────────────────────────────────────────────────────────

BUG #38 [HIGH] | Path Traversal em FileUtils ROM Import
Arquivo: app/src/main/java/com/vectras/vm/utils/FileUtils.java
Tipo: Validação inadequada de caminho de arquivo
Descrição: Se usuário fornece caminho como "../../../system/app/evil.apk" no
processo de importação de ROM, sem normalização e validação, arquivo pode ser
acessado/escrito fora da pasta designada.

Solução: Implementar File.getCanonicalPath() e verificar se resultado começa
com base path esperado. Rejeitar ".." e paths absolutos.

BUG #39 [HIGH] | Symlink Attack em VM Data Directory
Arquivo: app/src/main/java/com/vectras/vm/core/ProcessSupervisor.java (inferido)
Tipo: Symlink seguindo sem validação
Descrição: Se diretório de VM contém symlink para /data/data/com.vectras.vm
criado por aplicação maliciosa, acesso a "VM file" pode realmente acessar dados
sensíveis da app via symlink.

Solução: Usar File.getCanonicalFile() para resolver symlinks, verificar que
canonical path está dentro diretório esperado.

BUG #40 [MEDIUM] | Temporary File Leak em Setup Wizard
Arquivo: app/src/main/java/com/vectras/vm/setupwizard/SetupWizard2Activity.java
Tipo: Temporary files não deletados
Descrição: Arquivos temporários criados durante setup pode não ser deletados se
exception ocorre. Acumula em device storage over time.

Solução: Usar try-with-resources com File deletion ou implementar cleanup em
catch block.

BUG #41 [MEDIUM] | Directory Traversal em File List
Arquivo: app/src/main/java/com/vectras/qemu/utils/FileUtils.java
Tipo: Enumeração de arquivos sem validação
Descrição: Se enumerando arquivos em diretório fornecido pelo usuário, symlinks
podem causar traversal fora do diretório esperado.

Solução: Usar getCanonicalPath() para cada arquivo antes de processar.

Ciclo 6: Cryptography e Security Weaknesses
────────────────────────────────────────────────────────────────────────────

BUG #42 [HIGH] | Weak Randomness em Session Keys (Inferido)
Arquivo: app/src/main/java/com/vectras/vm/qemu/QmpClient.java (inferido)
Tipo: Random gerado sem SecureRandom
Descrição: Se gerando session IDs ou tokens para QMP, usar java.util.Random
ao invés de java.security.SecureRandom causa weak randomness.

Solução: Sempre usar SecureRandom para security-sensitive operations.
SecureRandom random = new SecureRandom();

BUG #43 [HIGH] | Hardcoded API Keys ou Secrets (Potencial)
Arquivo: app/src/main/java/com/vectras/vm/core/NativeFastPath.java (inferido)
Tipo: Hardcoded values em código
Descrição: Se qualquer valor secreto (chaves, tokens, hashes) estão hardcoded
em código fonte, decompilação de APK expõe imediatamente.

Solução: Mover secrets para BuildConfig encriptado, variáveis de ambiente,
ou servidor remoto.

BUG #44 [MEDIUM] | MD5 Hashing em Lugar de SHA256
Arquivo: app/src/main/cpp/vectra_core_accel.c (inferido)
Tipo: Weak hashing algorithm
Descrição: Se usando MD5 para integrity checking ou password hashing (improvável
mas possível), oferece apenas 128 bits de segurança e é quebrada.

Solução: Usar SHA-256 para hashing, bcrypt ou Argon2 para passwords.

BUG #45 [MEDIUM] | SSL/TLS Hostname Verification Bypassado
Arquivo: app/src/main/java/com/vectras/vm/network/RequestNetwork.java (potencial)
Tipo: SSL verification fraca
Descrição: Se implementando custom SSL context para conexões HTTPS, pode estar
ignorando hostname verification permitindo MITM attack.

Solução: Usar padrão HttpsURLConnection com verificação default, não customizar
TrustManager sem motivo válido.

BUG #46 [MEDIUM] | Plaintext Logging de Dados Sensíveis
Arquivo: app/src/main/java/com/vectras/vm/logger/VectrasStatus.java
Linhas: 324, 332, 250, 265
Descrição: Logs de status podem conter informações sensíveis (caminhos VM,
comandos QEMU, IPs de rede) que são visíveis em logcat para qualquer app.

Solução: Remover dados sensíveis de logs, usar BuildConfig.DEBUG para logs
debug-only, considerar criptografia de logs.

Ciclo 7: Serialization e Deserialization Issues

BUG #47 [MEDIUM] | Insecure Deserialization em Intent Extras
Arquivo: app/src/main/java/com/termux/app/TermuxActivity.java

Descrição: Se intent extras contêm Parcelable ou Serializable objects de fontes

implementar custom Parcelable.Creator com validação.

BUG #48 [MEDIUM] | JSON Parsing com Unsafe Defaults

Tipo: JSON deserialization sem validação
Descrição: Se parseando JSON de fontes externas sem schema validation, injection
ou malformed JSON pode causar crash ou unexpected behavior.

Solução: Usar try-catch ao parser JSON, validar tipos esperados, rejeitar
campos desconhecidos.

BUG #49 [LOW] | Properties File Injection
Arquivo: app/src/main/java/com/termux/app/TermuxPreferences.java
Linha: 180
Descrição: Se carregando properties de arquivo que pode ser modificado por
outro app, propósito properties pode ser alterado.

Solução: Validar file permissions (arquivo deve ser readable apenas por UID
app), usar encrypted preferences.

Ciclo 8: Fragment e Activity Lifecycle Issues
────────────────────────────────────────────────────────────────────────────

BUG #50 [HIGH] | Fragment Listener Not Unregistered
Arquivo: app/src/main/java/com/vectras/vm/main/vms/VmsFragment.java
Tipo: Listener não removido em onDestroy
Descrição: Se VmsFragment registra listener em onCreate, não remove em
onDestroy, listener retém fragment refence causando memory leak.

Solução: Implementar onDestroyView ou onDestroy que remove listeners.

BUG #51 [MEDIUM] | Dialog Not Dismissed in Activity Destroy
Arquivo: app/src/main/java/com/vectras/vm/setupwizard/SetupWizard2Activity.java
Tipo: Dialog leak
Descrição: Se exibindo ProgressDialog ou AlertDialog durante activity destroy,
dialog não é dismissed automaticamente causando window token error.

Solução: Implementar onDestroy que verifica e dismisses todos dialogs.

BUG #52 [MEDIUM] | Inner Class Activity Reference
Arquivo: app/src/main/java/android/androidVNC/VncCanvas.java
Tipo: Non-static inner class retains outer class reference
Descrição: Runnable anonymos em handler.post() mantém referência implícita à
Activity. Se handler é static ou long-lived, Activity leaks.

Solução: Usar static inner class com WeakReference a Activity, ou usar
HandlerThread ao invés de static Handler.

Ciclo 9: WebView, Intent e Data Security
────────────────────────────────────────────────────────────────────────────

BUG #53 [HIGH] | WebView XSS Vulnerability (Potencial)
Arquivo: app/src/main/java/com/vectras/vm/WebViewActivity.java (potencial)
Tipo: Cross-site scripting
Descrição: Se carregando conteúdo HTML de fontes externas em WebView sem
sanitização, JavaScript pode executar acessando dados da app.

Solução: Desabilitar JavaScript: webView.getSettings().setJavaScriptEnabled(false)
Se JavaScript é necessário, sanitizar HTML com library como jsoup.

BUG #54 [HIGH] | Intent Action Spoofing
Arquivo: app/src/main/AndroidManifest.xml
Tipo: Intent action não validada
Descrição: Se criando intent com ação que pode ser interceptada, app malicioso
pode interceptar e reedirecionar.

Solução: Usar explicit intents (especificar component) ao invés de implicit,
verificar component da resposta.

BUG #55 [HIGH] | PendingIntent Mutability
Arquivo: app/src/main/java/com/termux/app/TermuxService.java
Linha: 243
Descrição: PendingIntent criada sem PendingIntent.FLAG_IMMUTABLE (API 23+).
Aplicação maliciosa pode modificar intent causando ação diferente.

Solução: Usar PendingIntent.FLAG_IMMUTABLE | PendingIntent.FLAG_UPDATE_CURRENT.

BUG #56 [MEDIUM] | URI Grant Leak
Arquivo: app/src/main/java/com/vectras/qemu/utils/FileUtils.java
Tipo: URI permission grants não revogados
Descrição: Se compartilhando file via ContentProvider e grantUriPermissions=true,
permissões podem não ser revogadas causando data leak.

Solução: Usar revokeUriPermission ou implementar cleanup em Activity.onDestroy().

BUG #57 [MEDIUM] | Bundle Size Limits
Arquivo: app/src/main/java/com/termux/app/TermuxActivity.java
Tipo: Large bundle pode exceed size limits
Descrição: Se passando bundle grande entre Activities, pode exceder limit
causando TransactionTooLargeException.

Solução: Limitar tamanho de dados em bundle, usar application-level storage
para dados grandes (SharedPreferences, File).

BUG #58 [MEDIUM] | Broadcast Receiver Exported Without Permission
Arquivo: app/src/main/AndroidManifest.xml
Tipo: BroadcastReceiver exported sem permission check
Descrição: Se BroadcastReceiver é exported (visible a outras apps) sem
requerer permission, qualquer app pode enviar broadcast triggering ação.

Solução: Adicionar permission="..." ao receiver ou implementar
onReceive() que valida permissão do sender.

Ciclo 10: Thread Safety e Synchronization Issues
────────────────────────────────────────────────────────────────────────────

BUG #59 [HIGH] | Double-Checked Locking Anti-Pattern
Arquivo: app/src/main/java/android/androidVNC/RfbProto.java
Tipo: DCL anti-pattern com volatile
Descrição: Se usando if (field != null) { synchronized {...} } pattern sem
volatile, memory visibility issue pode ocorrer em multi-threaded access.

Solução: Usar synchronized throughout ou volatile com lazy initialization
via private static class holder.

BUG #60 [MEDIUM] | ConcurrentModificationException Risk
Arquivo: app/src/main/java/com/vectras/vm/logger/VectrasStatus.java
Linhas: 25-26 (Vector de listeners)
Descrição: Se iterating Vector<Listener> enquanto outro thread adiciona/remove,
ConcurrentModificationException é thrown.

Solução: Usar CopyOnWriteArrayList ou synchronized iteration com lock.

BUG #61 [MEDIUM] | Wait Without Loop Pattern
Arquivo: app/src/main/java/com/vectras/vm/core/ProcessSupervisor.java
Tipo: Object.wait() sem loop de verificação
Descrição: Se usar object.wait() sem while loop, spurious wakeups causam código
prosseguir sem garantia que condição foi satisfeita.

Solução: Usar while (condition) { lock.wait(); } ao invés de if.

BUG #62 [MEDIUM] | Volatile Field Ordering Issue
Arquivo: app/src/main/java/android/androidVNC/RfbProto.java
Tipo: Multiple volatile fields sem happens-before guarantee
Descrição: Se múltiplos volatile fields sendo atualizados, ordem de
visibility não é garantida sem synchronization explícita.

Solução: Usar synchronized block para múltiplas atualizações inter-relacionadas.
