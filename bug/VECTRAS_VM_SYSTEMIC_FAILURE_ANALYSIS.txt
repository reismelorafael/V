FALHA #1: allowBackup=true com Dados Sensíveis [ÓBVIO, IGNORADO]
────────────────────────────────────────────────────────────────────────────
Observação Óbvia: Se sua app gerencia máquinas virtuais e armazena imagens de
ROM, credenciais de rede, e configurações de usuário, qualquer desenvolvedor
competente identificaria IMEDIATAMENTE que permitir backup é negligência grave.

Que foi feito: NADA. Linha 42 do AndroidManifest.xml permanece:
android:allowBackup="true"

Mentalidade subjacente: "Backup é conveniente para desenvolvimento, vou deixar
para produção." FALSO. Negligência consciente.

Consequência sistêmica: Qualquer dispositivo com adb (desenvolvedor) pode
extrair dados completos. Uma ferramenta simples (adb backup) expõe TUDO.
Impacto: Não há recuperação. Dados foram comprometidos na instalação.

────────────────────────────────────────────────────────────────────────────

FALHA #2: Static Handler em Activity [ÓBVIO, IGNORADO]
────────────────────────────────────────────────────────────────────────────
Observação Óbvia: Android 101. Qualquer estudante de 3º semestre sabe que
static Handler referencing Activity causa memory leak irreparável.

Que foi feito: Código produção contém (X11Activity.java linha 99):
public static Handler handler = new Handler();

Que deveria ter sido detectado: Em linting inicial. Em code review. Em testes
de 2 horas em device real com profiler.

Mentalidade subjacente: "Funciona agora, não vejo crash." Negligência de
**pensamento sistemático sobre ciclo de vida**.

Consequência cascata: Activity criada → handler postDelayed → Activity
destruída → static reference sobrevive → próxima Activity reclama handler
antigo → memory leak progressivo → OOM crash após 10-15 rotações.

Não é um "bug à distância". É instalação sistêmica de **garantia de falha**.

────────────────────────────────────────────────────────────────────────────

FALHA #3: Race Condition em ProcessSupervisor Não Sincronizado [ÓBVIO, IGNORADO]
────────────────────────────────────────────────────────────────────────────
Observação Óbvia: volatile field acessado em múltiplas operações sem lock.
Campo é: private volatile Process process;

Operações não sincronizadas: destroy(), destroyForcibly(), waitFor()
Cenário: Thread A chama destroy(), Thread B simultaneamente chama
destroyForcibly() no mesmo Process object.

Resultado: double-free em native OS (processo descrito já destruído).
Mentalidade: "volatile garante visibilidade". FALSO. Volatile não sincroniza.

Impacto: VM não para limpa. Processo órgão no system. Sistema entra em estado
corrupto. Não há recuperação sem reboot device.

FALHA #4: Command Injection em SetupFeatureCore [ÓBVIO, IGNORADO]
────────────────────────────────────────────────────────────────────────────
Runtime.getRuntime().exec(cmdline);

Qualquer segurança 101: nunca passe string simples. Use array.

Que está em produção: String concatenation de inputs de usuário, passada
diretamente a shell sem quoting.

Mentalidade: "Usuários nossos são confiáveis." FALSO. Qualquer app pode
invocar intent. Qualquer malware pode fornecer input malicioso.

Impacto: Execute arbitrary commands como UID da app. Escape sandbox.
Acesso a dados de sistema. Não há remediação.

FALHA #5: Sem Verificação de Integridade de Arquivo [ÓBVIO, IGNORADO]
────────────────────────────────────────────────────────────────────────────
RomReceiverActivity aceita arquivo .cvbi de qualquer app com mimeType="*/*".

Nenhuma verificação de:
├─ Assinatura criptográfica
├─ Hash SHA-256
├─ Autoridade de origem
└─ Integridade de binário

Consequência: ROM malicioso injetado. Executado como binário ARM64.
Compromete VM completamente.

Mentalidade: "ROM é arquivo, é seguro na pasta." FALSO. Arquivo pode ser
construído por qualquer dev, lendo código em GitHub.
